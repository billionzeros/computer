# anton.computer — Update System

> How versioning and self-updates work across the desktop app, agent, and CLI.

---

## Overview

Three things update independently:

1. **Desktop app** — Tauri native app on the user's machine (updates via Tauri's built-in updater)
2. **Agent** — Binary running on the user's VM (self-updates via binary download)
3. **CLI** — Single `.mjs` file on the user's machine (self-updates via `anton update`)

All three share a single **manifest.json** that declares the latest version and download URLs.

## Version Resolution

The agent resolves its version at runtime:

1. `__AGENT_VERSION__` — injected at build time via esbuild `--define` (SEA binaries)
2. Walk up from compiled JS to find nearest `package.json` (dev mode)
3. Fallback: `"0.1.0"`

Git hash is resolved via `git rev-parse --short HEAD`, falling back to `"dev"`.

A `version.json` file in `~/.anton/` records deployment metadata (version, hash, deployer, timestamp) but is NOT used for version resolution — it's for diagnostics only.

## The Manifest

A single JSON file at repo root, served via GitHub raw URL:

```
https://raw.githubusercontent.com/billionzeros/computer/main/manifest.json
```

|-------|---------|
| `binaries` | Agent self-update |
| `sidecar_binaries` | CLI `anton computer setup` |
| `cli` | CLI self-update (`anton update`) |

## Agent Self-Update

### How the agent checks

The `Updater` class (`packages/agent-server/src/updater.ts`):

1. **On startup** — fetches manifest, compares versions via `semverGt`
2. **Every hour** — fetches again (`UPDATE_CHECK_INTERVAL`)
3. **On demand** — client sends `update_check` message

If a newer version exists:
- Caches the manifest in memory
- Includes `updateAvailable` in the `auth_ok` handshake
- Emits `update_available` event to connected clients

### How self-update works

Binary-only. No source/git fallback — if there's no binary URL for the current platform, the update fails with a clear error.

```
User clicks "Update"
        │
        ▼
Desktop sends: { type: "update_start" }
        │
        ▼
Agent runs binary update:
        │
        ├── 1. Download binary from release URL
        │      → { stage: "downloading" }
        │
        ├── 2. Atomic replace: backup current → rename new → chmod +x
        │      → { stage: "replacing" }
        │      (rolls back to backup on failure)
        │
        ├── 3. Write ~/.anton/version.json
        │
        ├── 4. systemctl restart anton-agent
        │      → { stage: "restarting" }
        │
        └── 5. Done (or error)
               → { stage: "done" }
```

Takes ~10-30 seconds. No git, pnpm, or build toolchain needed on the VM.

After restart, the desktop auto-reconnects and gets the new version in `auth_ok`.

### Binary path resolution

The updater finds the current binary in order:

1. `process.execPath` if it contains `anton-agent` (SEA binary)
2. `/usr/local/bin/anton-agent` (standard install)
3. `~/.anton/anton-agent` (fallback)

## Desktop App Updates

Separate system using Tauri's built-in updater plugin (`packages/desktop/src/lib/desktop-updater.ts`).

Checks `https://github.com/.../releases/latest/download/latest.json` (auto-generated by CI). Downloads `.dmg`/`.msi`/`.AppImage` and relaunches.

### ForceUpdateGate

If the desktop version is newer than the agent version (`semverGt(FRONTEND_VERSION, agentVersion)`), a full-screen blocking gate prevents app usage until the agent is updated. This prevents version skew where the UI expects features the backend doesn't have.

## Protocol Messages

All update messages use the **CONTROL channel** (0x00):

| Direction | Message | Purpose |
|-----------|---------|---------|
| Client → Agent | `update_check` | Check for updates now |
| Agent → Client | `update_check_response` | Current + latest versions, changelog |
| Client → Agent | `update_start` | Start self-update |
| Agent → Client | `update_progress` | Stage-by-stage progress (`downloading` / `replacing` / `restarting` / `done` / `error`) |

Plus one **EVENTS channel** message:

| Direction | Message | Purpose |
|-----------|---------|---------|
| Agent → Client | `update_available` | Proactive notification when periodic check finds a new version |

## Agent Distribution

| Form | What it is | When to use |
|------|-----------|-------------|
| **Bundle** (`anton-agent.mjs`) | Single JS file (~5MB), needs Node.js | Dev iteration (`make push`) |
| **SEA binary** (`anton-agent-linux-x64`) | Self-contained executable (~80MB), zero deps | Production (GitHub Release + self-update) |

Only linux binaries are built. macOS users run from source during development.

### Platform keys

| Key | Target |
|-----|--------|
| `linux-x64` | Linux x86_64 (most cloud VMs) |
| `linux-arm64` | Linux ARM64 (Graviton, Ampere) |

## Deployment Modes

| Command | What happens | Speed | Use when |
|---------|-------------|-------|----------|
| `make push` | Bundle locally → scp `.mjs` → restart | **~5s** | Day-to-day dev |
| `make sync` | Rsync source → build on VPS → restart | ~3 min | Need full source on VM |
| `make deploy` | Full Ansible playbook | ~10 min | First-time VPS setup |
| Self-update | Download SEA binary → replace → restart | ~15s | Production user updates |

## Release Workflow

```bash
make release    # Interactive: bumps versions, updates changelog + manifest, tags
git push origin main --tags    # Triggers CI
```

CI (`.github/workflows/release.yml`) runs on version tags (`v*`):
1. Builds agent SEA binaries for `linux-x64` and `linux-arm64`
2. Builds sidecar binaries (Go)
3. Builds CLI bundle (`anton-cli.mjs`)
4. Builds desktop app (macOS `.dmg`, Windows `.msi`, Linux `.AppImage`/`.deb`)
5. Creates GitHub Release with all artifacts + changelog
6. Updates `manifest.json` with git hash
